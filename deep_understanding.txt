1. EXECUTIVE SUMMARY & DOMAIN OVERVIEW
--------------------------------------------------------------------------------
BAR Editor (package name: `bar-editor`) is a specialized React 19 + Vite web application built to serve as an authoring, modding, static analysis, and packaging toolkit for "Beyond All Reason" (BAR) — a modern open-source Real-Time Strategy (RTS) game powered by the Recoil/Spring engine.

The core purpose of BAR Editor is to enable users to:
1. Parameterize & Modify Units: Introspect, override, and re-balance unit stats (HP, speed, metal/energy costs, build time, vision, radar, turn rates, weapon damages, ranges, accuracy, weapon swaps, custom parameters, etc.).
2. Clone & Create Unit Variants: Clone existing base units (including handling unit specific quirks like scavenger `scav_` and raptor `raptor_` prefixes), alter their weapon mounted slots, names, tooltips, build options, and visual icons.
3. Manage Build Menus & Rosters: Customise factory build menus, modify constructor build options, disable specific unit rosters, or apply preset roster packs (`BUILD_MENU_PACKS`).
4. Weapon Blueprints & Supporting Defs: Inject auxiliary weapon definitions, swap weapons across unit slots, and override weapon parameters directly inside unit weapon tables (`weapondefs`).
5. Death & Explosion Profiles: Customize explosion effects on death and self-destruct (`explodeas`, `selfdestructas`), adjusting damage, AOE, camera shake, and impulse factors.
6. Behavior Interceptors: Define custom Lua event hooks, targetable masks, state overrides, and tactical behaviors attached to units.
7. Package & Export Mod Options: Compile changes into clean, production-ready Lua tweak files (`custom_units.lua`, `tweakdefs`) and encoded Base64 lobby commands (`buildLobbyCommands`) for instant use in BAR multiplayer lobby setups without requiring game binary modifications.
8. Real-time Static Analysis & Compatibility Verification: Preflight-check tweak packages for Lua syntax errors, circular dependencies, parameter type mismatches, and lobby modoption buffer size limits before exporting.

2. SYSTEM ARCHITECTURE & TECH STACK
--------------------------------------------------------------------------------
• Core UI Framework: React 19, Vite 8 (ES Module mode), Vanilla CSS with an extensive CSS variable-based design system (`src/index.css`, `src/App.css`, `src/styles/`).
• Lua Parsing & AST Engine: `luaparse` (v0.3.1) for parsing Lua 5.1 ASTs from raw imported `tweakdefs`/`tweakunits` files.
• Image & Asset Pipeline: `sharp` (v0.34.5) for converting DDS textures/buildpics from upstream BAR repos into hash-deduplicated 192px WebP files.
• 3D & Graphics Visualizer: `three` (v0.185.1) for potential 3D asset inspection.
• Real-time Backend: Supabase (`@supabase/supabase-js` v2.110.2) supplying real-time workshop chat (`useTemporaryChat.js`) and online presence status (`useOnlinePresence.js`).
• Testing & Quality Assurance: Vitest (`vitest` v4.1.10) for unit/integration tests, Playwright (`@playwright/test` v1.61.1) for E2E tests, and Oxlint (`oxlint` v1.71.0) for high-performance JavaScript/JSX linting.

4. KEY DOMAIN CONCEPTS & DATA FLOWS
--------------------------------------------------------------------------------

A. PROJECT DOCUMENT STRUCTURE (`projectDocument.js`, `useProjectStore.js`)
   The entire state of a project is serialized into a normalized JSON document containing:
   - `meta`: Name, author, description, version, created/updated timestamps.
   - `unitOverrides`: Map of Unit ID -> modified parameter object (e.g. `{ health: 5000, metalcost: 300 }`).
   - `customUnitClones`: Array of clone specs:
     `{ id, baseId, newId, displayName, description, iconType, builderIds, weaponSwaps, addToOriginalBuilders }`
   - `buildMenuWizardSteps`: Array of step objects detailing additions/removals/orders to builder options.
   - `disabledUnitIds`: Array of unit IDs disabled from build rosters.
   - `tweakModules`: Raw pass-through Lua code modules provided by the user.
   - `deathExplosionTweaks`: Custom overrides for death/self-destruct explosion profiles.
   - `supportingWeaponDefs`: Auxiliary weapon definitions injected into units.
   - `unitCollections`: Hierarchical grouping trees for organizing unit batches.

B. LUA SERIALIZATION & COMPILATION PIPELINE (`tweakdefsHelper.js`, `tweakSerializer.js`)
   When building the final `tweakdefs` Lua script:
   1. `stripBlock()` removes old markers (`-- EDITP_BUILDMENU_BEGIN`, `-- EDITP_DEATH_PROFILES_BEGIN`, etc.) to prevent comment pollution.
   2. `generateClonesBlockLua()` generates helper Lua functions (`clone_copy`, `clone_set_name`, `clone_clean`, `clone_swap_weapon`) and dependency-sorts clones via topological sort (`sortClonesDependency`) to ensure base units exist before child clones are created.
   3. `generateBuildMenuBlockLua()` outputs Lua conditionals updating builder `buildoptions` arrays.
   4. `generateDeathProfilesBlockLua()` injects customized explosion weapon tables.
   5. `generateSupportingWeaponDefsBlockLua()` injects auxiliary weapon blueprints into `unit.weapondefs`.
   6. `serializeLuaTable()` formats JavaScript objects into valid, properly indented Lua syntax, handling numbers, strings, booleans, arrays, and sub-tables.

C. LOBBY COMMAND GENERATION (`lobbyModules.js`, `tweakSerializer.js`)
   - BAR start scripts receive custom tweaks via Base64-encoded lobby modoptions (`tweakdefs`, `tweakdefs1`..`tweakdefs9`, `tweakunits`..`tweakunits9`).
   - `encodeLobbyBase64()` converts Lua scripts to URL-safe Base64 without standard `+` characters (preventing start-script corruption of Lua operators like `~=`).
   - Split into chunks matching BAR lobby field size constraints, returning structured command strings ready to paste into BAR multiplayer lobby rooms.

D. COMPATIBILITY PREFLIGHT ANALYZER (`compatibilityPreflight.js`)
   Before export, the preflight engine inspects:
   - Project level validations (out-of-bound numbers, missing unit references).
   - Delivery limits (checking if encoded strings overflow the max available 9 numbered lobby slots).
   - Module parsing: Uses `luaparse` to check raw Lua syntax.
   - Dependency validation: Identifies circular dependencies, unresolved unit references, or lane crossing (e.g. `tweakdefs` referencing units created in `tweakunits`).
   - Asset paths: Flags unverified sound, model, or texture paths.

E. DATA PIPELINE & BUILD SCRIPTS
   - `sync_github_data.js`: Connects to Beyond All Reason upstream GitHub repos (`Beyond-All-Reason/Beyond-All-Reason`) to pull fresh unit definition files and update local metadata.
   - `sync-unitpics.mjs`: Downloads source DDS artwork, runs `sharp` to resize and convert to WebP, hashes output content for deduplication, and generates `unitpic-manifest.json`.
